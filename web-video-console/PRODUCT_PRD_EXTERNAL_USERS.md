# Codex Video Console 对外产品 PRD

## 1. 文档信息

- 产品名称：Codex Video Console
- 文档类型：产品需求文档 PRD
- 文档版本：v1.0 草案
- 面向对象：产品、设计、前端、后端、工作流工程
- 当前基础：`web-video-console/` 目录中的本地原型

## 2. 背景

当前的 `web-video-console` 已经验证了几个关键方向：

- 通过对话驱动视频任务创建
- 通过本地 session 延续上下文
- 通过账号 Profile 约束风格
- 通过模板约束结构
- 通过流程面板展示任务进度

这些能力对内部原型已经足够有价值，但如果后续要给其他用户使用，问题就不再只是“这个工具能不能跑”，而是要回答下面这些更产品化的问题：

1. 用户到底是谁
2. 用户如何稳定地产出结果
3. 系统的能力边界是否清晰
4. 多个账号、模板、session、任务之间的关系是否可理解
5. 非技术用户是否能安全使用

因此，这个前端后续不应被定义成“一个能连 Codex 的聊天网页”，而应被定义成：

**一个面向短视频生产的、账号驱动、模板引导、对话辅助、流程可视化的工作台产品。**

## 3. 产品愿景

打造一个让非技术运营也能稳定使用的视频生产工作台，用户可以：

- 先选择要服务的账号
- 再选择或锁定合适模板
- 通过自然语言输入选题或需求
- 清楚看到系统如何规划视频
- 在渲染前完成预览、审阅和修正
- 在不接触代码和本地目录的前提下完成产出

这个产品应同时具备四种属性：

1. AI 助手
2. 模板化视频生产器
3. 任务流程控制台
4. 审阅与确认工作台

## 4. 产品定位

### 4.1 核心定位

它不是通用聊天产品。  
它不是传统视频编辑器。  
它也不是无边界的 AI 生成器。

它的准确定位是：

**一个服务于短视频内容生产的结构化 AI 工作台。**

### 4.2 核心价值

通过把“账号风格”“模板结构”“AI 规划”“本地生产链路”组合到同一个工作台里，帮助用户更快地做出风格一致、结构稳定的视频。

可以用一个公式来概括：

`账号风格 x 模板结构 x 当前选题 x 指令微调 x 工作流自动化 = 可复用的视频产出`

### 4.3 产品承诺

用户在任何时刻都应该知道：

- 当前是给哪个账号做内容
- 当前使用的是什么模板
- 当前 AI 被允许改什么
- 当前任务处于什么阶段
- 最终会产出什么内容

## 5. 目标用户

### 5.1 主要用户：内容运营

用户特征：

- 负责一个或多个平台账号
- 有持续、稳定的短视频生产需求
- 通常不是开发者
- 更关注效率、风格一致性和可复制性

核心诉求：

- 快速出视频草稿
- 保持同一账号的风格稳定
- 尽量少写复杂 prompt
- 能看懂流程，不需要理解技术细节

### 5.2 次要用户：内容策略负责人 / 团队负责人

用户特征：

- 管理多个账号或多个运营
- 关注内容方向、风格纪律、输出质量
- 需要对账号风格和模板做治理

核心诉求：

- 可复用的账号 Profile
- 可控的模板体系
- 可管理的预设指令
- 清晰的任务历史和失败原因

### 5.3 系统管理员 / 工作流负责人

用户特征：

- 负责维护模板、流程、权限和执行链路
- 需要知道系统哪里容易出问题

核心诉求：

- 明确的权限边界
- 模板和账号的治理能力
- 可观察的日志和错误信息
- 稳定的执行链路

## 6. 产品目标

### 6.1 业务目标

1. 降低单条视频的生产时间
2. 提高同一账号输出的一致性
3. 降低对工程支持的依赖
4. 让新运营更快上手

### 6.2 用户目标

1. 在 5 分钟内得到一个可审阅的初稿
2. 不看底层日志也能理解系统当前状态
3. 能够基于账号风格重复生产内容
4. 能够继续历史 session，而不是每次从零开始

### 6.3 v1 产品目标

1. 支持账号驱动的任务创建
2. 支持模板选择与锁定
3. 支持 session 选择与延续
4. 支持预设指令与开放指令
5. 支持清晰的多步骤 pipeline 展示
6. 支持渲染前审阅

## 7. v1 不做什么

为了避免产品失焦，以下能力不建议纳入 v1：

1. 全自动平台发布
2. 多人实时协同编辑同一个任务
3. 浏览器内完整时间线编辑器
4. 面向最终用户的自由代码编辑
5. 任意本地目录浏览能力
6. 无限制的 prompt 执行能力

## 8. 产品原则

### 8.1 先选账号，再选模板

用户真正先要决定的是“谁在说话”，其次才是“用什么结构表达”。

原因：

- 账号决定人设、语气、节奏、CTA 和边界
- 模板决定画面结构、时长组织和 composition 路径

### 8.2 结构化控制优先于纯聊天

如果只有自由聊天，用户很难稳定控制产出。  
因此页面必须显式提供以下控制项：

- 账号选择
- 模板选择
- 指令预设
- session 选择
- pipeline 状态
- 审阅区

### 8.3 先审阅，再渲染

最终视频不应该像黑盒一样直接落地。

用户在渲染前应至少能够看到：

- Hook
- 正文结构
- 结尾 CTA
- 字幕风格
- 当前模板摘要
- 输出目标

### 8.4 默认安全

系统应默认假设用户不是开发者，因此不应要求用户理解：

- Codex CLI
- 本地工作目录
- prompt 拼装逻辑
- 脚本执行细节

### 8.5 一屏三心智区

页面结构应服务于三种不同思考动作：

- 左侧：提需求、继续对话
- 中间：配置任务、查看流程
- 右侧：审阅结果、准备确认

## 9. 核心业务对象

### 9.1 账号 Profile

定义：

账号 Profile 是一个可复用的内容身份对象，用于定义某个账号应该如何说话、如何结尾、适合什么模板、有哪些限制。

建议字段：

- 账号名称
- 平台
- 人设
- 风格标签
- 默认模板
- 默认 composition
- 默认时长
- 字幕风格
- CTA 风格
- 限制项

在产品中的作用：

- 提供稳定的内容身份
- 降低 prompt 歧义
- 保证同账号视频的一致性

### 9.2 模板 Template

定义：

模板是一个可复用的视频结构对象，绑定既定的 composition、画面节奏、镜头组织方式和风格规则。

建议字段：

- 模板 ID
- 模板显示名
- composition ID
- 适用平台
- 支持比例
- 推荐时长
- 风格标签
- 模板说明

在产品中的作用：

- 控制结构和渲染路径
- 降低生成波动

### 9.3 指令 Preset

定义：

一种可复用的指令增强层，用来在不重写完整 prompt 的情况下，对风格、节奏、结构或约束进行局部调节。

示例：

- 强化 Hook
- 更口语化
- 节奏更快
- 只改字幕
- 保留模板
- 更适合小红书

在产品中的作用：

- 降低用户写 prompt 的负担
- 提升修改效率

### 9.4 Session

定义：

一个持续的上下文容器，用于承接历史对话、上下文推理和任务连续性。

在产品中的作用：

- 延续上下文
- 避免用户重复解释背景
- 帮助同一任务反复细化

### 9.5 Job

定义：

一次具体的视频生产任务，包含账号、模板、session、用户请求、pipeline 状态和最终产物。

在产品中的作用：

- 是系统的最小生产单元
- 是审阅和追踪的基础单位

## 10. 要解决的关键问题

### 问题一：AI 输出不稳定

问题原因：

- 没有账号层约束
- 没有模板锁定
- 只有自由输入，没有显式控制

产品解法：

- 引入账号 Profile
- 支持模板锁定
- 显示当前生效的指令层

### 问题二：用户不知道系统在做什么

问题原因：

- 缺少面向用户的流程状态
- 技术日志过于底层

产品解法：

- 用人话展示 pipeline 步骤
- 提供可读的阶段状态和错误说明

### 问题三：用户分不清当前任务绑的是哪个 session

问题原因：

- 选中状态与实际绑定状态容易脱节
- session 概念在 UI 中不够明确

产品解法：

- 显示明确的 session 绑定状态
- 区分“已选择 session”和“已绑定 session”

### 问题四：同一主题需要生成多个账号版本

问题原因：

- 缺少账号风格层

产品解法：

- 把账号 Profile 提升为一级对象
- 让同一模板可以服务不同账号

## 11. 信息架构

### 11.1 全局结构

建议分为五层：

1. 工作区层
2. 账号与模板层
3. session 与对话层
4. 任务流程层
5. 预览与审阅层

### 11.2 页面组织方式

v1 建议保持单页工作台，不建议拆成多页面产品。  
原因是当前用户的行为高度连续、强依赖上下文，单页结构更适合维持任务状态。

建议通过以下轻量层级承载补充操作：

- 顶部栏
- session 抽屉
- 新增账号弹层或抽屉
- 模板详情面板
- 输出详情面板

## 12. 核心布局方案

### 12.1 顶部栏

作用：

- 建立工作区身份
- 展示系统可用性
- 提供全局设置

建议内容：

- 产品名称
- 当前工作区或项目名称
- Codex 连接状态
- 渲染服务状态
- 当前用户或角色
- 刷新按钮
- 设置入口

### 12.2 左侧栏：对话区

作用：

- 收集任务需求
- 延续上下文
- 提供继续修改的入口

建议模块：

- 当前 session 下拉框
- 新建对话按钮
- session 浏览入口
- 对话消息流
- 输入框
- 当前上下文 chips

交互要求：

- 对话列表整体可滚动
- 单条超长消息自身也可滚动
- 发送后必须立刻出现 pending 状态
- 失败时必须出现明确错误提示

### 12.3 中间栏：配置与流程区

作用：

- 承载所有显式可控的任务配置
- 展示任务状态与生成流程

建议模块：

- 当前任务摘要卡
- 账号选择器
- 新增账号入口
- 账号 Profile 摘要卡
- 模板选择器
- composition 显示
- 锁定模板开关
- 指令预设区
- 开放指令输入区
- 作用范围切换
- workflow / script / subtitle / resources / logs tabs
- 关键动作按钮

### 12.4 右侧栏：审阅区

作用：

- 帮助用户在渲染前后快速判断结果是否可用

建议模块：

- 手机预览框
- Hook / Body / Close 摘要
- 字幕风格摘要
- 输出文件状态
- 导出按钮
- 审阅提示

## 13. 详细模块需求

### 13.1 账号选择器

需求：

- 可搜索
- 显示账号名、平台、默认模板
- 在生成前可自由切换
- 在生成计划后切换时弹出确认

切换逻辑：

- 如果模板未锁定，则自动切换为账号默认模板
- 如果模板已锁定，则保留当前模板，但刷新账号限制和风格信息

### 13.2 新增账号

需求：

- 是显式入口，但不是默认自动弹出
- 必须支持关闭按钮、点击遮罩关闭、`Esc` 关闭
- 创建成功后立即写入账号列表

建议字段：

- 账号名称
- 平台
- 人设
- 风格标签
- 默认模板
- 默认时长
- CTA 风格
- 字幕风格
- 限制项

创建成功后的行为：

- 自动刷新账号列表
- 可选自动选中新账号

### 13.3 模板选择器

需求：

- 可搜索
- 显示模板名、composition、推荐时长、适用平台
- 支持自动模式和锁定模式

切换规则：

- 生成计划前，自由切换
- 生成计划后，系统提示：
  - 只切模板表现层
  - 重新生成计划
  - 取消切换

### 13.4 指令预设区

需求：

- 使用 chips 形式
- 选中态必须明确
- 建议按类别组织：
  - 风格类
  - 节奏类
  - 结构类
  - 约束类
  - 修复类

开放指令区：

- 多行输入框
- 占位符要明确引导用户
- 支持作用范围切换：
  - 仅本轮
  - 当前任务
  - 当前 session

### 13.5 Session 管理

需求：

- 顶部有当前 session 下拉框
- 有历史 session 浏览入口
- 绑定状态必须显式可见

建议状态：

- 未绑定
- 绑定中
- 已绑定
- 绑定失败

关键原则：

“已选中 session” 不等于 “当前任务已成功绑定该 session”。  
这两个状态必须在产品中清楚表达。

### 13.6 Pipeline 时间线

建议默认步骤：

1. 解析用户需求
2. 组装上下文与指令层
3. 生成视频计划
4. 生成脚本与字幕草稿
5. 生成 TTS 与时间轴
6. 绑定 composition 和资源
7. 执行渲染
8. 输出产物信息

每个步骤应支持状态：

- 待开始
- 执行中
- 已完成
- 失败
- 被阻塞

每个步骤建议展示：

- 简短中文说明
- 最近更新时间
- 失败摘要

### 13.7 审阅区

需求：

- 有预览图或播放器
- 有 Hook / Body / Close 摘要块
- 有字幕风格摘要
- 有输出文件状态
- 有导出能力

对于真正的外部用户版本：

- 不应直接暴露“打开本地目录”
- 应改为“下载产物”或“查看产物详情”

## 14. 关键用户流程

### 14.1 新用户首次使用

1. 用户进入产品
2. 看到推荐账号或样例账号
3. 选择账号
4. 选择模板或使用默认模板
5. 输入选题
6. 点击发送
7. 系统创建任务并绑定 session
8. 展示 pipeline
9. 用户审阅计划和预览

### 14.2 为已有账号创建视频

1. 选择账号
2. 确认模板
3. 选择预设指令
4. 输入主题
5. 发送消息
6. 审阅系统生成的计划
7. 通过补充指令继续细化
8. 生成配音
9. 执行渲染

### 14.3 延续历史 session

1. 打开 session 选择器
2. 选择历史 session
3. 确认绑定
4. 恢复上下文
5. 继续发送修改要求

### 14.4 任务中途切换模板

1. 用户修改模板
2. 系统检测当前任务是否已有计划
3. 弹出确认：
   - 仅替换表现层
   - 基于新模板重生成计划
   - 取消

## 15. Prompt 与控制逻辑

系统不应把用户输入原样直接发给底层模型。  
建议按以下顺序拼装上下文：

1. 系统工作流规则
2. 账号 Profile 规则
3. 模板规则
4. 当前启用的预设指令
5. 当前开放指令
6. 用户本轮输入
7. 历史 session 上下文
8. 当前任务状态

这样做的价值是：

- 输出更稳定
- 用户不用自己写复杂 prompt
- 账号与模板的控制力更强

## 16. 权限与安全设计

一旦给其他用户使用，这部分就不能再是“默认本地全权限”。

### 16.1 用户可见的安全原则

用户不应被允许：

- 任意浏览本地文件系统
- 执行任意命令
- 无权限修改生产模板
- 静默覆盖已有输出

### 16.2 建议角色模型

建议 v1 至少区分三类角色：

1. 运营者
   - 创建任务
   - 选择账号与模板
   - 使用预设指令
   - 查看并导出结果

2. 编辑 / 策略负责人
   - 创建和编辑账号
   - 管理部分预设指令
   - 审阅计划和风格

3. 管理员
   - 管理模板
   - 管理系统设置
   - 查看完整日志和执行状态

### 16.3 受限动作的表达方式

建议对“受限动作”使用显式禁用和说明，而不是直接隐藏。  
原因是用户需要理解系统边界，否则会把无法操作误判成系统异常。

## 17. 多用户产品化需要补充的决策

当前原型更接近本地单人工具。  
如果真的面向其他用户，需要尽快定下下面这些模型：

### 17.1 用户身份模型

需要明确：

- 用户是个人创作者
- 团队运营成员
- 还是外部客户

### 17.2 工作区模型

需要明确：

- 一个人一个工作区
- 一个团队一个工作区
- 还是未来多组织并存

### 17.3 Session 归属

需要明确：

- session 是否只属于创建者
- 团队负责人是否可查看团队成员 session
- session 是否允许复制

### 17.4 模板治理

需要明确：

- 谁能新建模板
- 谁能发布模板给其他人使用
- 模板是否要版本化

## 18. 成功指标

### 18.1 产品成功指标

- 从输入主题到拿到可审阅初稿的平均时间
- 渲染完成率
- 任务重试率
- 使用账号 Profile 的任务占比
- 使用模板复用的任务占比
- 无需工程介入即可完成的任务占比

### 18.2 交互健康指标

- session 绑定失败率
- 发送消息失败率
- 从计划到渲染的转化率
- 用户平均补充说明轮次

## 19. v1 范围建议

这个产品的正确 v1，不应该是“开放给所有人的通用 AI 视频平台”。  
更合适的定义是：

**一个单工作区、带权限、账号驱动的短视频生产控制台。**

建议 v1 必做范围：

1. 账号管理
2. 模板选择与锁定
3. session 延续
4. 预设指令与开放指令
5. pipeline 可视化
6. 审阅后再渲染
7. 导出产物

## 20. 路线建议

### 阶段一：受控内部 Beta

- 稳定前端交互
- 稳定 session 绑定
- 稳定 Codex bridge
- 定义账号 Profile 数据结构
- 定义模板元数据

### 阶段二：小团队版本

- 增加用户身份
- 增加权限模型
- 增加共享账号库
- 增加共享模板库
- 改善错误恢复能力

### 阶段三：运营成熟版

- 增加审核与批准流程
- 增加产物历史和审计记录
- 增加模板版本管理
- 增加素材复用能力

### 阶段四：更广泛的外部交付

- 增加自助 onboarding
- 增加组织级设置
- 增加计费、额度或资源配额
- 根据需要增加发布能力

## 21. 对前端设计的建议

### 21.1 保留三栏结构

三栏是合理的，因为它和用户的任务心智对应得上：

- 左侧是“说”
- 中间是“控”
- 右侧是“审”

### 21.2 减轻左侧栏负担

左侧应主要服务当前对话，不适合承载过深的 session 浏览结构。

建议：

- 保留当前 session 下拉框
- 将完整 session 浏览收进抽屉或弹层

### 21.3 强化中间栏

中间栏应成为真正的主工作区，而不是只显示状态。

建议：

- 把账号、模板、指令、流程全部集中在中间
- 补充 script、subtitle、resource 细化 tab

### 21.4 让右侧成为审阅面

右侧不应只是一个漂亮的手机框。

建议：

- 展示 Hook / Body / Close
- 展示字幕风格摘要
- 展示产物状态和下一步动作

## 22. 风险

1. 过度依赖自由聊天，导致控制力不够
2. 用户分不清账号、模板、session、job 四种对象
3. 本地执行链路不稳定，直接影响用户体验
4. 模板切换造成计划失效，但 UI 没解释清楚
5. 技术错误直接暴露给非技术用户

## 23. 待确认问题

下面这些问题会直接影响产品形态，建议尽快定下来。

### 战略问题

1. 未来用户主要是你的内部运营团队、代运营团队，还是外部客户？
2. 这个产品会先作为单团队工具存在，还是未来要支持多个组织？
3. 未来还是本地资源优先，还是希望逐步转成中心化服务？

### 工作流问题

4. 用户最常见的起点是：
   - 先选账号
   - 先选模板
   - 先写主题
   - 先继续历史 session
5. 模板是否允许高级用户自行创建？
6. 用户是否需要在 UI 里直接编辑脚本文本，而不是只通过聊天修改？

### 治理问题

7. 谁可以创建和编辑账号 Profile？
8. 一个账号 Profile 是否可以被多人共享？
9. 是否需要在渲染前增加审核步骤？

### 输出问题

10. v1 更重视“更快出初稿”还是“更强的一致性控制”？
11. 是否需要同一主题批量生成多个账号版本？
12. 是否需要“项目 / 活动”这一层来管理多个 job？

## 24. 下一步建议决策顺序

为了避免后面反复推翻设计，建议按这个顺序做产品决策：

1. 明确第一批真实用户是谁
2. 明确工作区和账号归属
3. 明确权限边界
4. 明确是继续本地优先，还是逐步服务化
5. 明确 v1 的成功指标

## 25. 总结

我认为这个产品最强、也最现实的方向是：

- 账号驱动
- 模板引导
- 对话辅助
- 流程可视
- 审阅后渲染

这个方向的好处是，它既保留了 AI 的灵活性，又不会把用户扔进不可控的黑盒里。对于非技术内容运营来说，这比“一个很强的聊天框”更像一个真正能交付的视频生产产品。
